DEVIATIONS FROM LAB DOC (read this first)
------------------------------------------
- Region: doc says us-east-1; this account's actual infrastructure
  (data-platform-vpc from Lab 1.2, the DataSync lab) lives in eu-west-1,
  so this stack targets eu-west-1 instead.
- State backend: rather than standing up a dedicated bucket, this
  stack's Terraform state reuses the shared data-lake-825765383386
  bucket + datasync-terraform-locks DynamoDB table already created by
  the DataSync lab (Lab 2.2), under key
  terraform/s3-data-lake-foundation.tfstate.
- CI/CD auth: static AWS credentials (AWS_ACCESS_KEY_ID/SECRET/
  SESSION_TOKEN as GitHub secrets), matching every other lab's pipeline
  in this account, rather than an OIDC-federated role. An OIDC bootstrap
  (IAM role + identity provider) was built first, then dropped — nothing
  else in this account relies on OIDC and it added setup steps this
  training sandbox didn't need.
- Security scanning: a Checkov CI gate was added temporarily to find
  real gaps against the initial build — CloudTrail had no SNS or
  CloudWatch Logs integration, no bucket in the stack had a lifecycle
  rule aborting incomplete multipart uploads, and the CloudTrail bucket
  itself wasn't access-logged. All of those were fixed. The gate itself
  was then removed (never part of the lab's scope), but the fixes it
  prompted were kept.
- IAM roles: DataEngineerRole, GlueServiceRole, RedshiftIAMRole are
  treated as pre-existing prerequisites from Lab 1.1 (iam-lab-terraform)
  — looked up via Terraform data sources, not created here. Confirmed to
  actually exist by reading that project's own deployed state file.
- CloudTrail governance beyond the lab doc: SNS notification on new log
  file delivery, real-time CloudWatch Logs integration, access logging
  on the CloudTrail bucket itself, lifecycle rules on the CloudTrail and
  logs buckets, and bucket-wide incomplete-multipart-upload cleanup.
  None of this is in the original lab doc; added as governance
  hardening at zero/near-zero additional cost.
- KMS encryption deliberately NOT used (SSE-S3 everywhere instead), to
  avoid the flat ~$1/month KMS CMK charge — a direct call made given
  this account's history of surprise billing.

Run 2 (2026-07-22 12:26, 2m9s):  FAILED — AccessDenied on S3 PutObject.
  The bucket policy denies PutObject unless the request explicitly
  carries x-amz-server-side-encryption: AES256; the folder-placeholder
  and test-data aws_s3_object resources didn't set that argument, so
  Terraform never sent the header.

  Fix: added server_side_encryption = "AES256" to both aws_s3_object
  resources (s3_folders.tf, test_data.tf).

COST ESTIMATE
--------------
At near-zero data volume (test data only, as of this deployment):
  S3 storage + requests:          negligible (<$0.01/month)
  CloudTrail management events:   free (first copy of management
                                   events is always free)
  CloudTrail S3 data events:      billed per 100,000 events — negligible
                                   at this activity level
  SNS notifications:              $0 (within free tier)
  CloudWatch Logs:                negligible (near-zero ingestion so far)
  No KMS CMK used (deliberate, avoids the flat $1/month charge)
Total estimated cost: <$1/month at current (test) data volume. Will
  scale with real S3 storage volume and CloudTrail S3-data-event volume
  once actual raw/ traffic starts flowing daily.
